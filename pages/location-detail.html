Location Detail

Everything tied to a single Location — status, current boat, checkpoints, notes, incidents, and tickets — with owner/contact/lease visibility governed by the same permissions used everywhere else those show up.

Purpose

Reached from Location list / map view, a checkpoint scan's context, or any link/preview card pointing at a location (chat, tickets, incidents). Acts as the hub for anything associated with this physical place, per the attachable-entities pattern in Data Model.

States & edge cases

  • Vacant slip, no boat — current-boat field shows an empty state rather than being hidden, so it's clear the slip is simply unoccupied, not that the field doesn't apply.
  • Location type without current_boat relevance — the field is omitted from the layout entirely for that type, not shown blank.
  • Owner exists but user lacks view_owner — the owner section doesn't appear at all; there's no indication to that user that an owner even exists, consistent with the permission's purpose.
  • Reservation upcoming but not yet active — shown as a distinct "upcoming" indicator next to (not replacing) the current status pill.

Mobile vs. desktop

Mobile: sections stack vertically and collapse by default (checkpoints, notes, incidents, tickets each expandable) to keep the initial view scannable. Desktop: sections are visible simultaneously in a wider layout, since there's room to show status, boat/owner info, and the associated-record lists without collapsing anything.

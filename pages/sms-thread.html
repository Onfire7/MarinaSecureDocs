SMS Thread

A single conversation with a Contact — full message history, plus a template picker for common replies.

Purpose

Reached from Comms Home or Missed Comms Detail. One thread per Contact-per-line, per Data Model — SMSThread; opening it marks it read.

Data shown

FieldSourceNotes
MessagesSMSMessage, ordered by timestampDirection, body, sender (for outbound), timestamp.
Thread contactSMSThread.contactName always shown; phone/email require view_contact.
Template pickerSMSTemplate, scope Global or Personal (owner = self)Global templates are marina-wide (managed in Admin → SMS Templates); Personal ones are managed by their owner from Comms.

Actions

ActionGatingBehavior
Send a messageplace_callsCalls the Twilio Function's POST /sms/send, optionally sourced from a template; the Function sends via Twilio and creates the outbound SMSMessage itself, per the Twilio-source-of-truth rule.
Pick a templateplace_calls (same gate as sending)Populates the composer with the template body, still editable before sending.
Manage personal templatesUnrestricted for one's own templatesAdd/edit/remove Personal-scope SMSTemplate records from here; Global templates are read-only from this screen.

States & edge cases

  • Offline — history already synced to InstantDB is fully viewable; sending is unavailable, since SMS is a Twilio (online-only) action — the composer reflects this honestly rather than silently queuing, per Architecture.
  • No place_calls — thread history is still viewable; only the composer/send action is hidden, matching Comms Home's view-vs-initiate split.
  • No templates configured — the picker is simply empty; typing a message from scratch is unaffected.

Mobile vs. desktop

Mobile: full-screen conversation, standard messaging-app layout. Desktop: can sit alongside Comms Home's thread list in a split view, similar to an email client.

Roles & Permissions

The trinary permission matrix editor — every role down one axis, every permission across the other, each cell an Allow / Deny / Undefined toggle.

Purpose

Reached from Admin Home, gated by manage_roles. This is the one screen that directly edits the mechanism described in Permissions — it doesn't compute effective permissions itself (that happens per-request everywhere else in the app), it just edits the per-role Allow/Deny/Undefined settings that feed that computation.

Data shown

FieldSourceNotes
RolesRole.nameColumns (or rows, depending on layout) across the matrix; marinas start with the five defaults (Security, Maintenance, Office, Marina Manager, Owner) and can add/duplicate/remove freely.
PermissionsThe fixed catalogEvery permission in Permissions, no more and no fewer — this screen can't add a new permission, only assign the existing ones.
Matrix cellsRole.permissions mapEach cell cycles through Allow (✓) / Deny (✕) / Undefined (blank), matching the trinary model exactly.

Actions

ActionGatingBehavior
Toggle a cellmanage_rolesCycles Allow → Deny → Undefined → Allow.
Add a rolemanage_rolesCreates a new Role with every permission defaulted to Undefined.
Duplicate a rolemanage_rolesCopies an existing role's full permission map as a starting point — e.g. a "Trainee" role cloned from Security, then denying a few permissions explicitly, per the worked example in Permissions.
Delete a rolemanage_rolesAny user still holding the deleted role simply loses whatever it granted; if it was their only role, they lose all role-gated access until reassigned.
Rename a rolemanage_roles

States & edge cases

  • Removing manage_roles from every role, including one's own — same lockout risk as deactivating the last manage_users holder on Users; should warn before allowing it, pending implementation-team confirmation.
  • Large matrix — with the full permission catalog (22 entries) and an arbitrary number of custom roles, the matrix can get wide; horizontal scroll with sticky role/permission headers is expected on smaller viewports.
  • Live effect — a permission change here takes effect immediately for every affected user's next action; there's no publish/staging step.

Mobile vs. desktop

Desktop is the primary surface — a matrix editor genuinely needs the screen width. Mobile falls back to a per-role, per-permission list view (pick a role, see/edit its permissions one at a time) rather than attempting the full grid.

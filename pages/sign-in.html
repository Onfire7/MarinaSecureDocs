Sign In

The entry point for any user with no active local session on this device — authenticates against the marina's InstantDB-compatible auth provider.

Purpose

Every deployment is a single marina's own subdomain (see Architecture), so there's no marina picker here — this screen only ever authenticates a User against the one InstantDB instance the subdomain already points to. There is no self-serve account creation: a user must already exist, created via Admin → Users by someone holding manage_users.

Data shown

FieldSourceNotes
Marina brandingMarinaSettingsName/logo, so the correct property is obvious before entering credentials.
Identifier fieldUser.email / User.phoneWhichever the chosen auth provider uses as the primary identifier — see open question below.

Actions

ActionGatingBehavior
Sign inNone (this is the gate)Submits the identifier plus whatever credential/challenge the auth provider requires; on success, establishes a local InstantDB session for that User and routes to the Dashboard.
Resend / forgot credentialNoneStandard recovery flow, provider-dependent.
Switch to a different userShown only when this device already has at least one signed-in sessionRoutes to User Switch instead of a fresh sign-in.
Open implementation detail. The exact auth provider and its challenge mechanism (password, magic link, OTP) is not yet finalized — see Architecture — Authentication. This page's field set will follow whatever that provider requires.

States & edge cases

  • First sign-in on a device requires connectivity — there's no local session yet for InstantDB to validate against offline. Once signed in, the session persists locally and later app use doesn't require this screen again.
  • Deactivated user (User.active = false) — fails with the same generic error as a wrong credential; the UI does not reveal that the account exists but is deactivated.
  • Already has a valid local session — this screen is skipped entirely on launch; the app opens straight to the Dashboard.

Mobile vs. desktop

Identical single-column form on both. No layout divergence — this is the one screen where a native-app feel and a webapp feel don't need to differ.

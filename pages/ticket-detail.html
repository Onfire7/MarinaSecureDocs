Tickets · Page Specification

Ticket Detail

Full view of one maintenance ticket — description, priority, status, assignment, and whatever it's attached to.

Purpose

Reached from Ticket queue, an Incident's linked-tickets list, a checklist item that raised it, or any attachment target's ticket list (Location, Boat, Asset, etc.).

Data shown

FieldSourceNotes
Title, descriptionTicket.title / descriptionDescription is rich-text (markdown).
Priority, statusTicket.priority / status
Created by / assigned toTicket.created_by / assigned_to
Attachment targetTicket.attached_toLinks to the relevant Location/Checkpoint/Boat/Contact/Asset detail page.
Source incidentTicket.source_incidentLink visible only with view_incidents — otherwise the ticket shows no indication it came from one, per Data Model.
Auto-generated flagTicket.auto_generatedShown with which maintenance rule triggered it, when applicable.
TimestampsTicket.created_at / resolved_at

Actions

ActionGatingBehavior
Take (self-assign)assign_ticket_to_selfOnly offered while unassigned.
Assign to someone elseassign_ticket_to_othersAvailable regardless of current assignment state — reassignment is still an assignment.
Change status / priorityNone — unrestrictedUpdating a ticket's own workflow state isn't gated the way assignment is; consistent with tickets generally being a low-friction, everyone-can-touch-them entity.
Add a noteUnrestrictedOpens New note dialog, pre-attached to the ticket's own attachment target — tickets themselves don't carry a separate Note relationship distinct from what they're attached to.

States & edge cases

  • No view_incidents, ticket has a source incident — the ticket is fully visible, just without any indication it originated from an incident; nothing about the ticket itself is hidden.
  • Reassigning an already-assigned ticket — allowed with assign_ticket_to_others; the previous assignee isn't notified specially beyond normal activity visibility.
  • Resolved ticket — resolved_at set when status reaches its terminal value (e.g. Complete); still editable afterward (status/priority changes aren't locked), unlike an Incident's author-lock behavior.

Mobile vs. desktop

Mobile: fields stack vertically, assignment and status controls as a sticky action bar. Desktop: fields and actions visible together in a wider single view.

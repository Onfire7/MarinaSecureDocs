Manual Check-In Dialog

Fallback path for when a checkpoint's NFC tag or QR code is missing, damaged, or unreadable.

Purpose

Reachable from the current tour view or checklist list ("Check in manually"). Requires a reason so a manual entry is always distinguishable from a scanned one in review — see Role Workflows.

Data shown / form fields

FieldTypeNotes
CheckpointSearch/selectDefaults to nearby checkpoints (by device GPS) when possible, full list otherwise.
ReasonRequired textCannot submit without it — the one field this dialog exists to collect.

Actions

ActionGatingBehavior
SubmitReason requiredCreates a Check-In record identical in shape to a scanned one, except method = Manual and reason set. GPS is still captured and validated against the checkpoint's radius the same way. Proceeds into whatever applicable checklist(s) exist, same as Checkpoint Check-In.
CancelNoneCloses without creating a record.

States & edge cases

  • No reason entered — submit stays disabled; this is the one hard validation rule on the dialog.
  • GPS unavailable (e.g. device location services off) — gps_at_checkin is left null and within_radius cannot be computed; the check-in still records, flagged for review rather than blocked.
  • Offline — writes locally like any other check-in.

Mobile vs. desktop

Primarily a mobile dialog, triggered in the field. Not a typical desktop path, but the same dialog works unmodified if triggered from a desktop context (e.g. a supervisor backfilling a check-in during review).

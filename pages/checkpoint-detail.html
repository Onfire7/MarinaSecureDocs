Locations · Page Specification

Checkpoint Detail

A read-oriented reference view of a single Checkpoint — its identity, GPS validation setup, associated checklists/tours, and recent check-in history — reached from the Location it belongs to.

Purpose

Distinct from two other checkpoint-related screens: Checkpoint Check-In is the guard-facing scan target (what the NFC/QR URL opens in the field), and Admin → Location Types & Locations setup is where a checkpoint's configuration is actually edited. This page is for looking a checkpoint up and reviewing its recent activity — e.g. an office user or manager checking "has this checkpoint been visited lately, and from where."

Data shown

FieldSourceNotes
Name & locationCheckpoint.name, Checkpoint.locationLocation links back to Location detail.
GUID URLCheckpoint.guid_urlCopyable link; a QR preview of the same URL is shown alongside it (the physical tag/code is generated from this, not stored separately).
GPS validation radiusCheckpoint.gps_validation_radius, falling back to MarinaSettings.gps_validation_radius_defaultShown with an indicator of whether this checkpoint overrides the marina default or inherits it.
Applicable checklist templatesCheckpoint.checklist_templatesEach links to Admin → Checklist Templates; shown with its trigger condition (time of day, season, etc.) so it's clear which applies when.
ToursCheckpoint.toursWhich Tours include this checkpoint, and each one's mode (Linear / Freeform / Randomized).
Recent check-insCheck-In records for this checkpointMost recent first: guard, timestamp, method (Scanned/Manual), and GPS validation result — an audit trail for patrol compliance review.

Actions

ActionGatingBehavior
Copy / share GUID URLNoneCopies the link — useful for re-printing a lost QR code via whatever process regenerates the physical tag.
Edit checkpoint configurationmanage_locationsRoutes to Admin → Location Types & Locations setup; this page itself has no inline edit form.
View full check-in historyNone beyond normal visibilityRoutes to Activity Log, pre-filtered to this checkpoint, for anything beyond the short recent list shown here.

States & edge cases

  • No check-ins yet — a new checkpoint shows an empty recent-activity state rather than an error.
  • No applicable checklist templates configured — shown plainly; a checkpoint is valid without one (a guard can still check in and just add a note/incident, per Checkpoint Check-In).
  • GPS radius override vs. default — always disambiguated in the UI so it's never ambiguous which value is actually in effect for spoof validation.

Mobile vs. desktop

Same content and layout on both — this is a reference/review screen rather than a field tool, so there's no strong mobile-specific interaction pattern to design for beyond the app's general responsive treatment.

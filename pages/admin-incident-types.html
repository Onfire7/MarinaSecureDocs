Incident Types

The (deliberately expandable) list of incident categories a marina uses — mostly for tidying up what already grows organically.

Purpose

Reached from Admin Home, gated by create_incidents — reused rather than introducing a dedicated permission, since the same permission already lets any user add a new type inline from New incident form, per Data Model — IncidentType. This screen exists mainly for cleanup: renaming a type, merging near-duplicates that accumulated from inline creation, removing ones that turned out unnecessary.

Data shown

FieldSourceNotes
NameIncidentType.name
Usage countCount of Incidents referencing this typeHelps decide what's safe to merge or remove.

Actions

ActionGatingBehavior
Add a typecreate_incidentsSame action available inline from New incident form; this is just the dedicated management view.
Rename a typecreate_incidentsExisting Incidents referencing it show the updated name, since it's a live reference, not a copied string.
Merge two typescreate_incidentsRe-points every Incident on the source type to the target, then removes the source.
Delete a typecreate_incidentsBlocked while any Incident still references it — must be merged or reassigned first.

States & edge cases

  • Type in use — delete is disabled, merge is offered instead as the safe path to removing it.
  • Duplicate-sounding names — not blocked or deduplicated automatically; this screen is the manual cleanup tool for exactly that situation.

Mobile vs. desktop

Same simple list-management layout on both; no meaningful layout divergence beyond the app's general responsive treatment.

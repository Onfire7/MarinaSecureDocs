Attachment-Target Picker

A shared sub-dialog, reused by Notes, Incidents, and Tickets whenever the attachment target isn't already implied by where the flow was opened from — search across all five valid target types and pick exactly one.

Purpose

Rather than each of New note dialog, New incident form, and New ticket form building its own target search, they all delegate to this one component when opened without a pre-filled context (e.g. from the Incidents nav item directly, rather than from a specific Location's detail page). Returns a single selection back to whichever flow opened it — it never creates or edits anything itself.

Data shown

FieldSourceNotes
Type filterFixed set: Location, Checkpoint, Boat, Contact, AssetMatches the attachment pattern in Data Model exactly — this picker is the one place that fixed list is enumerated as UI, everywhere else it's just "the attachment target."
Search resultsUnified fuzzy search across all five entity types, or scoped to one via the type filterEach result's visibility follows that entity type's own gating (e.g. a Contact result requires view_owner to appear) — the picker doesn't surface anything the user couldn't otherwise see.

Actions

ActionGatingBehavior
Select a resultSame as that result's own visibility gateReturns the selection to the calling flow (Note/Incident/Ticket form) and closes.
CancelNoneCloses with no selection; the calling flow's attachment target stays unset.

States & edge cases

  • No results — plain empty state, scoped to whichever type filter is active.
  • User can't see any of the five types — a theoretical edge case (would require lacking view_owner and having no unrestricted types visible, though Location/Checkpoint/Boat/Asset are themselves unrestricted to view) — in practice always has something to search, since only the Contact type carries a view gate here.
  • Reused across three different callers — the picker itself has no memory of which one opened it beyond returning to that flow; it's a stateless selection component.

Mobile vs. desktop

Mobile: full-screen search. Desktop: a smaller modal or inline dropdown, since there's more surrounding context already visible in the calling form.

Checklists & Tours · Page Specification

Tour Progress View

Security-only view of where a guard stands within their current Tour — presentation differs by tour mode, since Randomized mode must not reveal the full remaining route.

Purpose

Reached from the Tours tab on Checklist list. Read-only progress tracker — actual check-ins happen through Checkpoint check-in or the manual dialog, not here.

Data shown, by mode

ModeDisplay
LinearThe full ordered checkpoint list from Tour.checkpoints, with completed checkpoints marked and the next one highlighted.
FreeformThe full checkpoint set, marked visited/remaining, in no particular order — all must be visited before shift end but order is the guard's choice.
RandomizedOnly the currently-available next checkpoint(s) are shown, plus a remaining count. The rest of the set is deliberately withheld so patrol patterns stay unpredictable, per Role Workflows.

Actions

ActionGatingBehavior
Tap a checkpoint (Linear/Freeform)Security roleShows checkpoint detail/location; does not itself perform a check-in — that still requires the scan or manual flow.
View on mapSecurity roleShows the next checkpoint's GPS position relative to the guard's current location.

States & edge cases

  • No active tour — empty state directing the guard back to Checklist list or Dashboard.
  • Tour scope resets per shift — progress is derived from check-ins whose timestamps fall within the current Shift window (association by timestamp, not a direct reference, per Data Model — Shift), so a new shift starts every tour fresh.
  • Read access for managers — a manager reviewing a guard's live progress (e.g. via Reports) sees the same data read-only; not permission-gated in the fixed list, so treat as covered by general view_reports access.

Mobile vs. desktop

Mobile is the primary surface — used in the field mid-round. A desktop rendering is useful for a manager watching a guard's progress live from the office; same data, read-only either way, with desktop showing a larger map alongside the list.

Meter Update Dialog

Set an asset's current meter reading, or add accumulated hours if it has no meter at all.

Purpose

Reached from Asset detail. This is the direct, admin-style edit path — distinct from a Meter Reading checklist item, which captures the same underlying data (an AssetMeterReading) as part of normal checklist completion and is unrestricted, per Active checklist. Both paths feed the same maintenance-rule evaluation, per API Structure.

Data shown / form fields

FieldTypeNotes
Current valueRead-only, Asset.meter_reading or accumulated hoursShown for reference before entering a new value.
New readingNumber — only if has_meterAn absolute value (e.g. odometer mileage), not a delta.
Hours to addNumber — only if not has_meterAn incremental value, added to the running total rather than replacing it — matches the pages inventory's "set reading, or add hours for meterless assets."
Reason for lower readingText, conditionally requiredAppears only when the entered value is lower than the asset's current reading; must be filled before Save is enabled. Stored as AssetMeterReading.correction_reason.

Actions

ActionGatingBehavior
Savemanage_assetsCreates an AssetMeterReading (source = Manual entry on asset page) and updates Asset.meter_reading. Meter-based maintenance rules are evaluated inline as part of this same write — a due rule creates a Ticket immediately, not on a delay.
CancelNoneCloses without recording a reading.

States & edge cases

  • New reading lower than current (for a metered asset) — shows a warning inline and requires the Reason field to be filled before Save is enabled; not a hard block on the scenario itself (meter replacement/correction is a real case), just a forced explanation, matching the pattern already established for the "outside radius" GPS flag elsewhere: allow, but record why.
  • Meterless asset — the form only ever offers "add hours," never "set an absolute value," since there's no absolute reading to set.
  • Maintenance rule triggers on save — the resulting Ticket is auto_generated = true, identical in every other respect to a manually created one; see Ticket detail.

Mobile vs. desktop

Same dialog on both; no meaningful layout divergence beyond the app's general responsive treatment.

Checkpoint Check-In Screen

What opens when a guard taps their phone against an NFC tag or scans a QR code at a checkpoint — the checkpoint's own guid_url.

Purpose

This URL is the checkpoint, functionally: it's bound 1:1 to a Checkpoint record and is the only way this screen is reached. On load it immediately logs the visit and surfaces whatever's relevant right now, per Role Workflows — Checking in at a checkpoint.

On-load sequence

  1. Capture device GPS.
  2. Create a Check-In record: checkpoint, user (current session), timestamp, method = Scanned, gps_at_checkin, and within_radius computed against Checkpoint.gps_validation_radius (falling back to MarinaSettings.gps_validation_radius_default).
  3. Evaluate Checkpoint.checklist_templates against their trigger conditions (time of day, season, etc.) to find which apply right now.
  4. Present the matching checklist(s) for completion, and/or a place to add a note or log an incident even if none apply.

Data shown

FieldSourceNotes
Checkpoint name & locationCheckpoint.name, Checkpoint.location
GPS validation resultCheck-In.within_radiusShown only if flagged false — otherwise the check-in proceeds silently.
Applicable checklist(s)Checkpoint.checklist_templates, trigger-evaluatedZero, one, or more.

Actions

ActionGatingBehavior
Begin an applicable checklistMust be signed inRoutes into Active checklist for that instance.
Add a note / log an incident with no checklistcreate_incidents for incidents; notes unrestrictedOpens the relevant dialog, pre-attached to this checkpoint.

States & edge cases

  • Not signed in — the URL still opens (it's a public deep link), but routes to Sign In or User Switch first, then resumes the check-in flow.
  • GPS outside the checkpoint's radius — the check-in still records (within_radius = false); it isn't blocked, but is flagged for later review, matching the data model's spoof-guard-not-spoof-block design.
  • Multiple applicable checklists — presented as a short list to work through in sequence rather than merged into one.
  • Offline scan — the Check-In write succeeds locally and syncs on reconnect, identical to any other InstantDB write.

Mobile vs. desktop

Mobile-only in practice — this screen exists to be opened by a phone tapping an NFC tag or scanning a QR code in the field. No meaningful desktop variant is designed; if opened on desktop (e.g. testing a checkpoint's URL) it renders the same content without any field-specific chrome.

Checkpoint Check-In Screen

What opens when a guard taps their phone against an NFC tag or scans a QR code at a checkpoint — the checkpoint's own guid_url.

Purpose

This URL is the checkpoint, functionally: it's bound 1:1 to a Checkpoint record and is the only way this screen is reached. On load it immediately logs the visit and surfaces whatever's relevant right now, per Role Workflows — Checking in at a checkpoint. GPS capture happens in the background rather than gating any of this — see below.

On-load sequence

GPS acquisition is the one step here that can take anywhere from under a second to several seconds depending on signal, so it's deliberately taken off the critical path — the guard is never staring at a spinner waiting for it:

  1. Create a Check-In record immediately: checkpoint, user (current session), timestamp, method = Scanned, with gps_at_checkin and within_radius both left null/pending. This is a local-first InstantDB write, so it completes practically instantly regardless of connectivity.
  2. Evaluate Checkpoint.checklist_templates against their trigger conditions (time of day, season, etc.) to find which apply right now.
  3. Present the matching checklist(s) for completion, and/or a place to add a note or log an incident even if none apply — the guard can start working immediately, before GPS has resolved.
  4. In parallel with steps 2–3, request device GPS. Once it resolves, update the same Check-In record with gps_at_checkin and within_radius (computed against Checkpoint.gps_validation_radius, falling back to MarinaSettings.gps_validation_radius_default).

GPS as a background update

While GPS is pending, a small, unobtrusive indicator (e.g. a subtle pill near the header — generic "Saving…" copy, not "Saving location…"; see note below) signals that something is still happening, without blocking any interaction: checklist items, notes, and incidents all work normally while it's showing. Once GPS resolves, the indicator clears — silently if the check-in lands within radius, or replaced by the same "outside radius" flag described under Data shown if it doesn't. If GPS never resolves within a reasonable timeout (location services disabled, unusually poor signal), the indicator settles into an "unavailable" state rather than spinning indefinitely; gps_at_checkin and within_radius simply stay null, which is an accepted terminal state, not an error.

Indicator copy stays generic. Background-save indicators like this one should say only that something is still saving ("Saving…"), not what — never name the specific field or subsystem (e.g. not "Saving location…" or "Saving GPS…"). This keeps the pattern reusable verbatim for any other field that ends up saving asynchronously in the background, without a spec-by-spec rewrite of the copy.

Avoiding duplicate check-ins & tab pileup

Because this screen is reached by re-opening the same URL every visit, two related failure modes need explicit handling: a reload accidentally logging a second check-in, and — on a device where the app isn't installed — old check-in tabs piling up over the course of a shift.

Duplicate check-in on reload

Immediately after step 1 creates the Check-In record, the tab rewrites its own URL via history.replaceState to include that record's ID (e.g. ?checkin=<id>), without a full navigation. Any subsequent load of this same tab — a manual refresh, the OS resuming a backgrounded tab — sees that ID and resumes the existing Check-In and its checklist state instead of creating a new one. As a safety net for cases where that URL state doesn't survive (a different device, the param getting stripped), the write path also checks for an existing Check-In for this checkpoint + user within the last few minutes before creating a new one, resuming it if found rather than duplicating.

Tab pileup (not installed as a PWA)

When the app is installed, an NFC tap or QR scan reopens the existing app window rather than spawning a new browser tab, which avoids this problem at the root. Where it isn't installed, each tab instead tries to close itself once it's no longer needed, rather than being left to accumulate:

  • On completion — once every Checklist this check-in triggered (Checklist.triggered_by referencing it) reaches Complete, or, if none applied, once the guard has had the chance to add a note/incident, the tab calls window.close().
  • On a 10-minute idle timeout — if the tab has been open more than 10 minutes with nothing left unfinished, it attempts the same close.
  • On reactivation, as a fallback — browsers only allow a script to close a tab it also opened, and a tab launched by an OS-level NFC/QR intent may or may not qualify depending on platform. Where window.close() is silently refused, the same completion/idle check re-runs every time the tab regains focus, so a tab that couldn't close itself right away closes the next time the guard actually looks at it — old tabs cascade closed one at a time as they're encountered, rather than needing a manual mass-cleanup.
  • If never revisited — a tab that can't close and is never reactivated simply redirects to the Dashboard once its completion/idle condition is met, so anything left behind at least shows something current rather than a dead checkpoint screen.

States & edge cases

  • Not signed in — the URL still opens (it's a public deep link), but routes to Sign In or User Switch first, then resumes the check-in flow.
  • GPS outside the checkpoint's radius — once GPS resolves, the check-in record updates to within_radius = false; it isn't blocked (the check-in already exists and the guard may already be mid-checklist), but is flagged for later review, matching the data model's spoof-guard-not-spoof-block design.
  • GPS never resolves — after a reasonable timeout, the pending indicator settles to "unavailable" and the Check-In record simply keeps gps_at_checkin/within_radius null. The check-in itself is still valid; it's just unvalidated, same as a manual check-in with no GPS (see Manual Check-In Dialog).
  • User navigates into a checklist before GPS resolves — expected and unblocked by design; the pending indicator follows along (or the resulting flag appears) regardless of which sub-screen the guard has moved to, since it's tied to the Check-In record, not this particular screen.
  • Multiple applicable checklists — presented as a short list to work through in sequence rather than merged into one.
  • Offline scan — both the initial Check-In write and the later GPS update succeed locally and sync on reconnect, identical to any other InstantDB write; GPS itself is a device capability and doesn't require connectivity to resolve.
  • Reload before the URL rewrite lands — a reload in the brief window before history.replaceState completes falls back to the time-window dedupe check rather than the URL check; either path resumes the same Check-In instead of duplicating it.
  • Tab won't close — if the browser refuses window.close() (a tab it didn't itself open), it isn't retried in a loop; it waits for the next focus/visibility event or falls back to redirecting to the Dashboard, per "Avoiding duplicate check-ins & tab pileup" above.

Mobile vs. desktop

Phone is the primary case — tapping an NFC tag or scanning a QR code in the field — but the screen is built responsively rather than mobile-only: NFC-capable tablets and hybrid laptops are a real, if less common, way this gets opened (a guard using a rugged tablet for rounds, or a hybrid laptop with NFC at a fixed post that still moves between checkpoints). It is not designed for a permanently-placed desktop workstation, since nothing about a fixed desktop ever taps an NFC tag or scans a QR code — that case is only reachable by pasting the URL directly (e.g. testing a checkpoint's link), and renders the same content without any field-specific chrome. Layout scales from a single-column phone view up through tablet/hybrid-laptop widths using the same breakpoint approach as the rest of the app (see Architecture — Frontend), rather than assuming one fixed viewport size.

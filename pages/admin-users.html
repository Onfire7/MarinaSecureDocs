Admin · Page Specification

Users

Invite, edit, deactivate, and assign roles to every user at the marina — gated by manage_users.

Purpose

The only way new users enter the system, per Sign In — there's no self-serve account creation. Inviting a user here provisions them in both Clerk and as an InstantDB User record, linked by clerk_user_id, per Architecture — Authentication.

Data shown

FieldSourceNotes
Name, email/phoneUser.name / email / phone
RolesUser.rolesShown as tags; a user can hold multiple.
ActiveUser.activeDeactivated users are visually distinct and excluded from assignment pickers elsewhere in the app, but stay listed here for historical reference.

Actions

ActionGatingBehavior
Invite a usermanage_usersCollects name and email/phone, triggers Clerk's invitation flow, and creates the corresponding User record once accepted.
Edit a usermanage_usersName, contact info, role assignment.
Deactivate / reactivatemanage_usersToggles User.active; does not delete the record, preserving it for historical Activity Log entries, past assignments, etc.

States & edge cases

  • Deactivating the last manage_users holder — blocked outright. The check is computed the same way effective permissions are (per Permissions): if no other active user would be left holding manage_users through any role, the deactivation is rejected with an explanation, not just warned about.
  • Removing all roles from a user — allowed; that user simply loses access to every role-gated feature until reassigned, without being deactivated outright.
  • Pending invitation not yet accepted — shown with a distinct status, separate from active/deactivated.

Mobile vs. desktop

Mobile: a searchable list. Desktop: a sortable table with role tags and status visible without opening each user — this is squarely an office/desk workflow, so desktop is the primary case here.

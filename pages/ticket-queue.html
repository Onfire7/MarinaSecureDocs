Ticket Queue

Every maintenance ticket, sorted by priority — the shared worklist for Maintenance, and anyone else who needs to see what's open.

Purpose

Reached from the Tickets top-level nav item. Viewing the queue has no dedicated permission — every role can see it, matching ticket creation being unrestricted by design (see Permissions). What's gated is acting on a ticket: taking one for yourself, or handing one to someone else.

Data shown

FieldSourceNotes
Title, priority, statusTicket.title / priority / statusSorted Urgent → Low by default; status set is admin-adjustable (see Admin).
AssigneeTicket.assigned_toBlank for unassigned tickets.
Attachment targetTicket.attached_toLocation, Checkpoint, Boat, Contact, or Asset — shown as a short label/link.
SourceTicket.source_incident, Ticket.auto_generatedA small tag when the ticket came from an Incident (link gated by view_incidents) or was auto-generated by a maintenance rule.

Actions

ActionGatingBehavior
Open a ticketNoneNavigates to Ticket detail.
Create a ticketNone — unrestricted, per PermissionsOpens New ticket form.
Take an unassigned ticketassign_ticket_to_selfSets assigned_to to the current user directly from the queue, no need to open the ticket first.
Assign to someone elseassign_ticket_to_othersInline assignee picker from the queue.
Filter by priority / status / assigneeNoneClient-side filter over the subscribed query.

States & edge cases

  • No queue-visibility permission needed — worth calling out explicitly since it's the exception rather than the rule elsewhere in the app: there is deliberately no view_tickets permission in the fixed catalog.
  • Neither assignment permission — the queue is still fully visible and creation still works; only the inline take/assign actions are hidden.
  • Empty queue — a genuinely good state, shown plainly rather than as an error.
  • Auto-generated ticket — auto_generated = true for anything created by a maintenance rule (see Data Model — Asset); shown identically to a manually created ticket otherwise.

Mobile vs. desktop

Mobile: a single priority-sorted list, filters as a collapsible sheet. Desktop: a sortable table with filters always visible and inline assignment without opening each ticket.

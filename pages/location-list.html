Locations · Page Specification

Location List / Map View

Browse the marina's full location hierarchy — docks, slips, buildings, RV sites, whatever the marina has defined — as a flat filterable list or as pins on a real-world map.

Purpose

The landing point for the Locations top-level nav item. A list/map toggle over the same underlying set of Location records, filtered by LocationType and status. This screen is for finding and navigating to a location, not for defining the location hierarchy itself — creating location types and locations happens in Admin → Location Types & Locations setup, gated by manage_locations.

This is the geographic pin map, not the schematic map view. The map mode described in this spec plots each Location's real-world gps_coordinates on a live map (pins) — the same coordinates used for checkpoint GPS validation. A second, schematic map view is also planned for this screen: an admin-uploaded layout image (MarinaMap) with each Location plotted as a color-coded rectangle (LocationMapPlacement) — the same underlying capability the Reservation calendar/list uses for its own map view, just with an occupancy/type overlay here instead of a reservation-status one. See Data Model and the note under Reservations in Page Specifications. That view mode is still to be designed and added to this page; the geographic pin map described below doesn't depend on it.

Actions

Change a location's statusmanage_locationsInline quick-edit from the list (e.g. marking a slip Out of Service) — reuses the same permission as full location editing rather than introducing a separate status-change permission, per Permissions.
Add a locationmanage_locationsShortcut into Admin → Location Types & Locations setup, where creation actually happens.

States & edge cases

  • No locations defined yet — empty state; for a user with manage_locations, it prompts straight into Admin setup, otherwise a plain "nothing here yet" message.
  • Location with no GPS coordinates — present in list mode, silently excluded from map mode rather than shown mislocated.
  • Deep hierarchy — nesting depth isn't fixed (a type's valid_parent_types can chain arbitrarily); the list view supports drilling down (parent → children) rather than assuming a flat two-level dock/slip structure.
  • Current boat changes — reflects live; when a boat moves slips (see Data Model — Boat), this list updates immediately via InstantDB subscription, with the move itself preserved in the Activity Log rather than a location history table.

Mobile vs. desktop

Mobile: list is the default view; hierarchy is drill-down (tap Dock C to see its Slips) rather than a long flat list, and map opens full-screen when toggled. Desktop: list and map can sit side by side, with filters always visible and the full hierarchy browsable without drilling down screen by screen.

Reports · Page Specification

Reports Home

The landing point for the Reports top-level nav item — gated entirely by view_reports.

Purpose

Per Role Workflows, a Marina Manager/Owner leans on this section day to day for shift summaries, incident history, ticket throughput, and call logs — an oversight view across the marina, distinct from the Activity Log's raw chronological feed.

Data shown

SectionSourceNotes
Shift reportsShift, most recent firstGuard, date, report_sent_at status; each links to Shift report.
Incident historyIncident, aggregatedCounts and trends by type/status over a selected date range — numbers only, no individual incident titles/details. Visible under view_reports alone, regardless of view_incidents, per the aggregate-data rule below.
Ticket throughputTicket, aggregatedCreated vs. resolved counts over a selected date range; tickets have no dedicated view permission anyway, so this is unrestricted the same way Ticket queue is.
Call logsCall / SMSThread, aggregatedVolume counts only, over a selected date range — no caller identity, no message content, no individual entries. Visible under view_reports alone, regardless of view_calls/view_sms; individual call/text detail still lives in Comms, gated there as normal, not duplicated here.

Actions

ActionGatingBehavior
Open a shift reportview_reportsNavigates to Shift report.
Adjust date rangeview_reportsApplies to the aggregate sections; shift reports are browsed by list, not range-filtered the same way.

States & edge cases

  • No view_reports — the Reports nav item doesn't appear; this page is unreachable, matching the same all-or-nothing pattern as Incident list's view_incidents gate.
  • Aggregate data vs. detail data — the rule for this screen: a section that only ever shows counts/trends (no individual record, no name, no content) is gated by view_reports alone, full stop — that's Incident history, Ticket throughput, and Call logs as specified above. Anything that would show an individual record — a specific incident's title, a specific caller's name, a specific message — falls back to that record's own normal gate (view_incidents, view_calls, view_contact, etc.) and is hidden entirely without it, same as Activity Log feed. This screen currently has no such detail sections, only aggregates — if one is added later (e.g. a "most active incidents" drill-down list), it must follow the detail rule, not the aggregate one.
  • No shifts yet — empty state for a brand-new marina.

Mobile vs. desktop

Mobile: sections stack vertically, each collapsible. Desktop: sections arrange in a grid so several are visible at once, similar to the Dashboard's card layout.

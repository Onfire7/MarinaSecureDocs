Activity Log Feed

A chronological, filterable feed of nearly every database change at the marina — immutable, generated at write time, and scoped per entry to whatever permission governs its subject.

Purpose

Reached from the Activity Log top-level nav item. There's no single "view activity log" permission; per Role Workflows, a manager's Activity Log is "scoped to whatever they have permission to see." Concretely: an entry whose subject is an Incident requires view_incidents to appear; a Call requires view_calls; a Contact requires view_owner; and so on down the same permission each entity type already uses elsewhere. Entries about unrestricted entities (Tickets, Locations, Reservations, Boats, Assets, Notes) are visible to everyone.

Actions

ActionGatingBehavior

States & edge cases

  • Entry subject the user can't see — omitted from the feed entirely, not shown redacted — the same "omit, don't partially reveal" rule used everywhere else in the app.
  • Entry about the Activity Log itself — never occurs; per Architecture, the Activity Log doesn't log its own writes.
  • High-frequency, low-value changes — excluded at write time per marina/entity-specific rules (see Architecture — Activity Log generation), not filtered out here; this feed shows everything that was actually logged.
  • Retention purge — entries older than the marina's configured window disappear from this feed once the scheduled purge runs, except Protected ones, which persist indefinitely.

Mobile vs. desktop

Mobile: a single filterable chronological list. Desktop: filters always visible alongside a denser list, similar to Ticket queue and Incident list.

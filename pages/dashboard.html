Home · Page Specification

Dashboard

The Home landing screen — a customizable set of cards. Roles determine a sensible default arrangement; each user can then show, hide, and reorder any card they have access to.

Purpose

There's no single fixed "home" layout, but there's also no free-for-all: a card is only ever available to add if the user holds whatever permission already governs the data that card displays. Roles come into it only as a starting point — each role ships with a default set of cards, shown in a default order, reflecting what that role most needs to see immediately. A user is free to depart from that default (hide a card they don't care about, add one they're permissioned for but whose role doesn't default to it, reorder the whole stack); the role-derived arrangement is just what a new or unconfigured user sees first. See Role Workflows for the narrative behind each card.

Card permissions reuse data permissions — no card-specific gates. A card's visibility requirement is exactly whatever permission already controls the underlying data it reads (e.g. Incident summary requires view_incidents because it reads Incident records — the same gate that applies everywhere else Incidents are shown). No new "can see this card" permission is introduced for any card; where the fixed permission catalog has no dedicated view-gate for a card's data (tickets, reservations — see Permissions), the card itself requires none, and some cards (a clock) read no gated data at all and are simply always available.

States & edge cases

  • No cards available at all — shouldn't occur in practice (the Clock card requires no permission and is always offerable), but the fallback is a minimal view showing only account info and nav.
  • Permission revoked for a currently-shown card — if a role change removes the permission a visible card depends on, that card drops out of the layout automatically on next load; it isn't left visible-but-broken, and it isn't restored automatically if the permission is later regranted (the user re-adds it, or resets to role defaults).
  • New role granted — does not retroactively rearrange an already-customized layout; the newly-available cards simply become addable. Only a fresh, never-customized user (or an explicit "reset to role defaults") picks up a role's default arrangement wholesale.
  • Shift ended offline — the dashboard reflects ended_at immediately (local write); the report itself sends once connectivity returns and the scheduled backstop sweep picks it up, per Architecture.
  • Multiple active tours/checklists — the shift card summarizes only; full detail lives on Checklist list and Tour progress.

Mobile vs. desktop

Mobile: cards stack vertically in the user's saved order (defaulting to role order for an uncustomized user), with the Shift card pinned to the top for Security by default. Desktop: cards arrange in a grid so several are visible without scrolling, still following the same saved order/visibility — see Architecture — Frontend for the general mobile/desktop divergence principle. Show/hide/reorder controls are reachable from an edit mode on both, though drag targets are naturally larger on mobile.

Marina Settings

The single settings record that lets one codebase serve very differently-run properties — GPS defaults, retention, recording, phone lines, and the shift report distribution list.

Purpose

Reached from Admin Home, gated by manage_marina_settings. Edits the one MarinaSettings record per marina, per Data Model — there's exactly one, not a list.

Data shown / form fields

FieldSourceNotes
GPS validation radius defaultMarinaSettings.gps_validation_radius_defaultFallback used by any Checkpoint without its own override, per Location Types & Locations setup.
Activity Log retentionMarinaSettings.activity_log_retention_daysEntries older than this are eligible for purge by the activity-log-retention Scheduled Function unless flagged Protected, per API Structure.
Call recording / transcription togglesMarinaSettings.call_recording_enabled / call_transcription_enabledGoverns whether Active Call Panel and call history ever show a recording/transcript at all.
Shift report recipientsMarinaSettings.shift_report_recipientsEmail distribution list, per Architecture — Shift report delivery. An inline warning appears directly beside this field, and only here, when it's empty — see States & edge cases below.
Phone linesMarinaSettings.phone_linesEach line's Twilio number, label, and routing behavior (e.g. a Security line skipping straight to a security menu).
Allow overlapping reservationsMarinaSettings.allow_overlapping_reservationsOff by default. Governs whether New Reservation Form blocks a booking whose dates overlap an existing one for the same target, or permits it (e.g. for a marina running a waitlist/overbooking model).

Actions

ActionGatingBehavior
Edit any fieldmanage_marina_settingsTakes effect immediately — there's no publish/staging step, matching Roles & Permissions' live-effect behavior.
Add / edit / remove a phone linemanage_marina_settingsAdding a line requires the underlying Twilio number to already exist on the marina's Twilio account; this screen configures the app's record of it, not Twilio itself.

States & edge cases

  • Lowering the retention window — takes effect on the next scheduled purge, not retroactively instantaneously; existing entries older than the new window are purged (unless Protected) at that next run, not the moment the setting is saved.
  • Removing the last phone line — leaves the marina with no Twilio-routed number; calling/texting features have nothing to route through until a line is reconfigured. Should warn before allowing, pending implementation-team confirmation.
  • Empty shift report recipients — the shift-report Twilio Function still runs and compiles the report, it simply has nowhere to send it. This is surfaced as an inline warning next to the field on this screen only — not as a banner elsewhere in the app, not on Reports or Shift report, and not blocking save. It's a nudge at the one place it's actually configured, nothing more.

Mobile vs. desktop

Same form layout on both; this is an infrequent configuration task, not a field or high-frequency workflow, so no special mobile treatment beyond the app's general responsive behavior.

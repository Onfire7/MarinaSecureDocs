Checklists & Tours · Page Specification

Checklist List

Every checklist currently relevant to the signed-in user — assigned to them directly or to a role they hold — plus, for Security, a Tours tab.

Purpose

The landing point for the Checklists top-level nav item (see Introduction — Navigation). Surfaces anything generated by a trigger the user needs to act on: clock in/out, scheduled time, checkpoint visit, incident type, or manually started.

Data shown

FieldSourceNotes
Template nameChecklist.template → ChecklistTemplate.name
StatusChecklist.statusNot Started / In Progress / Complete.
Triggered byChecklist.triggered_byShown as a short label, e.g. "Checkpoint: Dock C Gate" or "Scheduled 6:00 AM."

Actions

ActionGatingBehavior
Open a Not Started / In Progress checklistMust be the assigned_to user, or hold the role it's assigned toOpens Active checklist.
Open a Complete checklistSame as aboveOpens Checklist detail (read-only).
Filter by statusNoneClient-side filter over the subscribed query.
Tours tabSecurity roleSwitches to a checkpoint-set view backed by Tour progress; not shown to users without the Security role.

States & edge cases

  • Empty state — no checklists currently apply; distinct copy for "nothing assigned right now" vs. "you hold no roles with checklist assignments."
  • Personal templates — a checklist from a Personal-visibility template only ever appears for its creator, never for others holding the same role.
  • Role-restricted templates assigned to a role, not a user — any user holding that role sees the same unclaimed instance until someone opens (and implicitly claims) it.

Mobile vs. desktop

Mobile: a single vertical list, most urgent/oldest In Progress items first, status filter as horizontal chips. Desktop: a sortable table with the Tours tab rendered as an adjacent panel rather than a full tab switch, since there's room to show both at once.

Incidents · Page Specification

Incident List

Every logged incident, gated entirely by view_incidents — the one entity in the app where visibility itself, not just some sub-detail, is a hard permission gate.

Purpose

Reached from the Incidents top-level nav item, which is itself only shown to users holding view_incidents (see Permissions). Unlike tickets, there is no "everyone sees the list, some actions are gated" split here — without the permission, incidents don't exist as far as the UI is concerned.

Data shown

FieldSourceNotes
Title, type, statusIncident.title / type / statusStatus: Open / Under Review / Resolved / Closed, plus a marina's Custom option.

AssigneeIncident.assigned_toNullable — assignment is optional on incidents, unlike tickets.

Actions

ActionGatingBehavior
Open an incidentview_incidentsNavigates to Incident detail.
Log a new incidentcreate_incidentsOpens New incident form.
Filter by status / typeNone beyond view_incidentsClient-side filter over the subscribed query.

States & edge cases

  • No view_incidents — the Incidents nav item itself doesn't appear; this page is unreachable, not shown-but-empty.
  • create_incidents without view_incidents — a legal but unusual combination: this user could log an incident (via a direct link from a checklist, say) without being able to browse the list afterward. Rare in practice, since most roles that create incidents also need to see them.
  • Empty list — plain empty state; a marina with no open incidents is a good sign, not an error.

Mobile vs. desktop

Mobile: single filterable list. Desktop: sortable table with filters always visible, matching the pattern used on Ticket queue.

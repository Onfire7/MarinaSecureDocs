Reports · Page Specification

Shift Report

View a compiled end-of-shift report, and resend it if needed.

Purpose

Reached from Reports home. Shows the same content that was (or will be) emailed to the marina's MarinaSettings.shift_report_recipients, per Architecture — Shift report delivery.

Data shown

FieldSourceNotes
Guard, shift windowShift.guard / started_at / ended_at
Delivery statusShift.report_sent_atNull means not yet sent — either still within the shift, or waiting on the scheduled backstop sweep if the shift ended offline.
Compiled contentCheck-ins, checklists, incidents, and tickets whose timestamps fall within the shift's windowSame compilation logic used for the actual email — association by timestamp, not a direct Shift reference, per Data Model — Shift.

Actions

ActionGatingBehavior
ResendThe shift's own guard, or view_reportsCalls the same POST /reports/shift/:shiftId/send Netlify Function used for the original on-demand send (see API Structure) — it compiles fresh from current data and hands off to the Twilio Function to actually send via SendGrid and update report_sent_at.

States & edge cases

  • Not yet sent — shown with a pending indicator rather than a blank delivery-status field; the content itself is still fully viewable, compiled live from current data regardless of whether the email has gone out.
  • Resend while offline — disabled, since the send path ultimately depends on Twilio SendGrid (an online-only action, per Architecture); the report content remains viewable offline.
  • Content changes after the shift ends (e.g. a late incident comment) — viewing this page always compiles fresh, so it can show more than what the original email contained; a resend captures that current state, not the original snapshot.

Mobile vs. desktop

Same read-only layout on both; no divergent interaction pattern beyond the app's general responsive treatment.

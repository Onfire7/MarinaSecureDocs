Comms · Page Specification

Chat Room

One internal, topic-based conversation — entirely separate from calls and SMS, with no database relationship linking the two.

Purpose

Reached from Comms Home. Per Role Workflows — Comms: chat rooms, any user can start a room and invite specific users and/or roles into it; this page is where that room's conversation actually happens.

Data shown

FieldSourceNotes
Title / topicChatRoom.title / topic
MessagesChatMessage, ordered by timestampAuthor, body, attachments, timestamp.
Entity preview cardsChatMessage.body, parsed for Location/Asset/Contact/etc. URLsRendered inline for display only — pasting a URL is link parsing, not a stored relationship, per Data Model.
Invited users / rolesChatRoom.invited_users / invited_rolesinvited_roles is a live reference to Role, not a snapshot — a user granted that role after the room was created gains access to it immediately, without being re-invited.

Actions

ActionGatingBehavior
Send a messageMust be invited (directly or via role), or hold view_all_chatsCreates a ChatMessage; a view_all_chats holder who isn't actually invited can read but sending as a non-participant is a judgment call left to the marina's own norms, not a hard technical gate beyond visibility.
Attach a file/imageSame as sendingStored on ChatMessage.attachments.
Paste an entity URLSame as sendingAuto-renders a preview card; purely cosmetic, no relationship written.
Manage inviteesRoom creatorAdd/remove invited users or roles after creation; not gated by a new permission — ownership of the room is the gate, consistent with avoiding permission bloat (see Permissions).

States & edge cases

  • Not invited, no view_all_chats — the room isn't reachable or listed anywhere; there's no "request access" path.
  • Offline — chat is ordinary InstantDB data, not a Twilio action, so it works fully offline (send, read, attach) and syncs on reconnect, unlike SMS.
  • Role membership changes mid-room — a user removed from an invited_roles role loses access to the room going forward, but past messages they sent remain (their authorship doesn't get retracted).

Mobile vs. desktop

Mobile: full-screen single room. Desktop: room list and active room can sit side by side, matching typical chat-app conventions.

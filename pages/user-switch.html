User Switch

Fast handoff on a shared device — a tablet mounted at the security desk, used by whoever is on shift — without a full sign-out/sign-in cycle.

Purpose

Shift changes happen daily and need to be fast. This screen exists so a shared device can hold multiple users' local sessions at once and let the incoming user pick themselves from a short list rather than typing full credentials every handoff. Personal devices never see this screen in practice — they only ever hold one user's session (see Architecture — Authentication).

Actions

ActionGatingBehavior
Select a listed userNonePrompts a lightweight re-authentication challenge (mechanism depends on the chosen auth provider — see note below), then makes that user's session active and routes to the Dashboard.
Add another userNoneRoutes to Sign In for a user not yet listed on this device; on success, that user is added to the device's list for future switches.
Remove a user from this deviceNone (self) / manage_users (removing someone else's session)Clears that user's local session from the device entirely — used when someone leaves the marina or a device is being repurposed.
Open implementation detail. The exact re-authentication challenge for switching (PIN, biometric, short password) is not yet finalized — it's a requirement on whichever InstantDB-compatible auth provider is chosen, not a design decision made here. See Architecture — Authentication.

States & edge cases

  • No users yet on this device — the list is empty and the screen effectively becomes Sign In (or presents it directly).
  • Switching mid-active-checklist — the outgoing user's in-progress local state is per-user, so it's preserved and simply not shown while another user's session is active; it reappears on their own checklist list next time they switch back in.
  • Offline switch — works fully offline once a user has previously authenticated on this device, since the local session and InstantDB's local data are already present.

Mobile vs. desktop

Shared devices are overwhelmingly tablets mounted at a fixed post, so this screen is designed tablet-first — large tap targets, a short vertical list. The same screen serves a desktop shared workstation identically; there's no meaningfully different desktop layout to design for here.

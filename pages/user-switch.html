User Switch

Fast handoff on a shared device — a tablet mounted at the security desk, used by whoever is on shift — without a full sign-out/sign-in cycle.

Purpose

Shift changes happen daily and need to be fast. This screen is the UI for Clerk's multi-session support: a shared device can hold several users' authenticated Clerk sessions simultaneously, and the incoming user picks themselves from a short list rather than typing full credentials every handoff — switching is just calling Clerk's setActive() against their existing session. Personal devices never see this screen in practice — they only ever hold one active Clerk session (see Architecture — Authentication).

Data shown

FieldSourceNotes
Users with an active session on this deviceClerk's client-side session listEveryone currently signed in on this device via Clerk's multi-session state, most recently active first.
Name / avatarUser.name (resolved from Clerk's clerk_user_id)

Actions

ActionGatingBehavior
Select a listed userNoneCalls Clerk's setActive() for that user's existing session — no credential re-entry, since the session is already authenticated — then routes to the Dashboard.
Add another userNoneRoutes to Sign In for a user not yet listed on this device; on success, Clerk adds their session to the device's multi-session list for future switches.
Remove a user from this deviceNone (self) / manage_users (removing someone else's session)Ends that user's Clerk session on this device entirely — used when someone leaves the marina or a device is being repurposed.

States & edge cases

  • No users yet on this device — the session list is empty and the screen effectively becomes Sign In (or presents it directly).
  • Switching mid-active-checklist — the outgoing user's in-progress local state is per-user, so it's preserved and simply not shown while another user's session is active; it reappears on their own checklist list next time they switch back in.
  • Offline switch — works fully offline once a user has previously authenticated on this device: Clerk's existing session and InstantDB's local data are already present, so no network round-trip is needed to switch.

Mobile vs. desktop

Shared devices are overwhelmingly tablets mounted at a fixed post, so this screen is designed tablet-first — large tap targets, a short vertical list. The same screen serves a desktop shared workstation identically; there's no meaningfully different desktop layout to design for here.

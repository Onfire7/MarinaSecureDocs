Reservations · Page Specification

Reservation Detail

Full record of one reservation — contact, target, dates, and (for Public reservations only) the billing fields.

Purpose

Reached from Reservation calendar/list, or from the target Location/Asset's own detail page (an upcoming reservation shown alongside current status, per Data Model).

Data shown

FieldSourceNotes
ContactReservation.contactName always shown; phone/email require view_contact.
TargetReservation.targetLinks to Location detail or Asset detail, whichever applies.
StatusReservation.statusRequested / Confirmed / Checked In / Checked Out / Cancelled.
Expected / actual datesexpected_checkin/checkout, actual_checkin/checkoutActual dates are nullable until check-in/out happens, per Reservation Check-In / Check-Out Dialog.
Early check-in / late check-outReservation.early_checkin / late_checkoutShown only when the target's reservation_visibility is Public.
Rate / deposit / balanceReservation.rate / deposit / balanceSame Public-only gating — entirely absent from the layout for an Internal reservation, not shown as zero/blank.

Actions

ActionGatingBehavior
Edit reservation detailsmanage_reservationsCovers contact, dates, rate/deposit/balance (where applicable) — anything short of actually checking in/out.
Override status/datesmanage_reservationsManually setting status or dates outside the normal check-in/out flow (e.g. correcting a mistake) — explicitly called out in Permissions as part of this permission's scope.
Check in / check outmanage_reservationsOpens Reservation Check-In / Check-Out Dialog rather than editing actual_checkin/actual_checkout inline, since that action also has to update the target's status.
Cancelmanage_reservationsSets status to Cancelled; does not delete the record.

States & edge cases

  • Internal reservation — the whole billing section (early/late fields, rate/deposit/balance) is omitted from the layout, not shown empty, per reservation_visibility.
  • No manage_reservations — the reservation is still fully viewable (subject to view_contact for contact detail); every edit/check-in-out action is hidden.
  • Cancelled reservation — remains visible in history; a target's status/upcoming-indicator no longer reflects it once cancelled.
  • Asset target — identical layout to a Location target, just linking to Asset detail instead; the reservation model doesn't distinguish the two beyond the target reference itself.

Mobile vs. desktop

Same layout on both — fields stack vertically on mobile, sit in a wider single view on desktop; no divergent interaction pattern beyond the app's general responsive treatment.

Asset Detail

Full record of one asset — status history, meter, maintenance rules, and checkout log.

Purpose

Reached from Asset list, a Ticket/Incident/Note attached to it, a Reservation targeting it, or a Meter Reading checklist item's asset link.

Data shown

FieldSourceNotes
Name, categoryAsset.name / category
Current statusAsset.current_status, with full AssetStatusLog history below itLatest value displayed prominently; the log is the running record it's derived from.
MeterAsset.meter_reading / meter_type, or accumulated hours if meterlessRecent AssetMeterReading history shown alongside, with each entry's source (Manual / Checklist item).
Maintenance rulesAsset.maintenance_rulese.g. "every 3,000 miles," "every 90 days since last completed"; edited in Admin, shown read-only here.
Checkout logAssetCheckout, most recent firstOnly shown for checkoutable assets; current open checkout (if any) highlighted at the top.
Reservation configAsset.reservation_enabled / reservation_visibility / post_return_statusRead-only summary here; upcoming reservations shown alongside status per Data Model — Reservation.

Actions

ActionGatingBehavior
Log a status updatemanage_assetsCreates a new AssetStatusLog entry; updates Asset.current_status.
Check out / returnmanage_assetsOpens Checkout / return dialog — checkout records are explicitly part of this permission's scope in Permissions, so "any role can check an asset in or out if permitted" (per Role Workflows) means "if they hold manage_assets," not a separate gate.
Update the metermanage_assetsOpens Meter update dialog. Distinct from a Meter Reading checklist item, which is unrestricted as part of normal checklist completion (see Active checklist) — this is the direct, outside-of-a-checklist edit path.
Add a noteUnrestrictedOpens New note dialog, pre-attached.
Log an incidentcreate_incidentsOpens New incident form, pre-attached.
Create a ticketUnrestrictedOpens New ticket form, pre-attached.

States & edge cases

  • Checkout doesn't itself change status — creating an AssetCheckout record doesn't automatically write an AssetStatusLog entry; if the marina wants "checked out" reflected as a status, whoever processes the checkout logs that separately. Only return has a modeled automatic status effect (post_return_status).
  • Asset already checked out — the checkout action becomes a return action instead; a checkoutable asset can't have two simultaneous open checkouts (a second time_in-null record for the same asset).
  • Not checkoutable — the checkout log section and action are both omitted entirely.
  • No maintenance rules configured — shown plainly; an asset with a meter still works fine with zero rules, it just never auto-generates tickets.

Mobile vs. desktop

Mobile: sections stack vertically, collapsible. Desktop: sections visible together in a wider layout, matching the pattern used on Boat detail and Location detail.

Tickets · Page Specification

New Ticket Form

Log a maintenance issue — available to every role by default, with no permission gate.

Purpose

Reached from Ticket queue, an Incident detail page ("raise a ticket"), a failed checklist item (Verify Task rejection or Door Check double-mismatch), or any attachment target's detail page. Per Permissions, ticket creation is deliberately unrestricted — every role can create one, including Office logging an issue an owner called in.

Data shown / form fields

FieldTypeNotes
TitleText, required
DescriptionRich text (markdown), optional
PrioritySelect — Low / Medium / High / UrgentDefaults to Medium.
Attachment targetLocation / Checkpoint / Boat / Contact / AssetPre-filled when opened from a specific record's context; otherwise uses the shared Attachment-target picker.
Source incidentHidden field, set automaticallyPopulated only when opened from an Incident's "raise a ticket" action — not user-editable.

Actions

ActionGatingBehavior
SubmitNone — unrestrictedCreates the Ticket with created_by = current user, status = Open, auto_generated = false. Routes to Ticket detail on success.
CancelNoneCloses without creating a ticket.

States & edge cases

  • Opened with no attachment context — the attachment target is required before submitting; there's no such thing as an unattached ticket, per Data Model.
  • Opened from a checklist item's failure — title/description/attachment target are pre-filled from that context (e.g. the checkpoint's Location, a note about which Door Check failed), still editable before submitting.
  • Assignment isn't offered here — a new ticket is always created unassigned; taking or assigning happens afterward from Ticket queue or Ticket detail, gated normally by assign_ticket_to_self/assign_ticket_to_others.

Mobile vs. desktop

Same form on both; no meaningful layout divergence beyond the app's general responsive treatment.

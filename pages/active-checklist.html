Active Checklist

Item-by-item completion of an in-progress Checklist instance — the most interaction-heavy screen in the app, since it hosts all five item-type behaviors.

Purpose

Reached from Checklist list, a checkpoint scan/manual check-in, or a Location-Based Check item opening a nested instance. Walks the user through Checklist.items (via ChecklistItemResult) in order, recording a result per item and writing it locally-first (see Architecture — Database) — nothing here requires connectivity.

Item-type behavior

TypeInteractionWrites
Simple CheckTap to mark complete.result = boolean true; completed_at set.
Verify TaskConfirm done, or reject with a written reason, or reject and immediately raise a linked ticket. If the template's config flags it, a rejection first prompts the guard to attempt the task themselves before finalizing the rejection.result = Confirmed / Rejected-Reason / Rejected-Ticket; linked_ticket set if a ticket was raised.
Door CheckSelect observed state. Mismatch vs. expected → prompted to correct (open/close/lock/unlock) and select again. Second mismatch → note or ticket required; the app will not let a mismatch pass silently.result includes observed state plus retry history; linked_ticket set if raised on second mismatch.
Location-Based CheckOpens a nested sub-checklist scoped to a specific Location; returns to the parent item list on completion.Nested Checklist instance, same schema, referenced from the parent item's result.
Meter ReadingSelect the asset (fixed by the template, or user-selected at runtime — e.g. "select your patrol vehicle") and enter its current value.Creates an AssetMeterReading (source = Checklist item) and updates Asset.meter_reading; may trigger a maintenance-rule Ticket inline on write, per API Structure.

States & edge cases

  • Offline throughout — every write here is local-first; leaving and resuming later (even after reconnecting) preserves exact progress with no special handling.
  • Ticket creation from a failed item — the resulting Ticket.source_incident stays null (this isn't an incident-sourced ticket) but ChecklistItemResult.linked_ticket connects it back; see Data Model.
  • Abandoning mid-checklist — status stays In Progress indefinitely; it simply reappears on Checklist list until finished.

Mobile vs. desktop

Mobile: one item presented full-screen at a time, swipe/tap to advance — matches the "phone on a dark dock" usage pattern. Desktop: the full item list is visible at once with each item expanding inline, since a desk-bound user reviewing or completing a checklist benefits from seeing the whole sequence.

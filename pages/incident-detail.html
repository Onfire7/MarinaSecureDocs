Incident Detail

Full record of one incident — its original content, an addendum thread that stays open forever, and any tickets it spawned.

Purpose

Reached from Incident list, a linked ticket (with view_incidents), or a checklist item that raised it. Two distinct edit windows apply here, and getting them right matters: the incident's own content locks to its author at end of shift, but comments never lock for anyone holding create_incidents, per Role Workflows.

Data shown

FieldSourceNotes
Title, type, detailsIncident.title / type / detailsDetails is optional rich-text (markdown).
StatusIncident.statusOpen / Under Review / Resolved / Closed, plus Custom.
Author, assigneeIncident.author / assigned_toAssignee is optional.
Attachment targetIncident.attached_toLinks to the relevant Location/Checkpoint/Boat/Contact/Asset.
AddendumsIncidentComment, ordered by created_atAuthor, body (markdown), timestamp — a permanent, append-only thread.
Linked ticketsIncident.linked_ticketsEach links to Ticket detail.

Actions

ActionGatingBehavior
Edit the incident's original contentMust be the author, and only until the end of their shiftAfter that window, the original content locks regardless of who's viewing — including the author themselves.
Add a comment/addendumcreate_incidentsAppends an IncidentComment; available forever, to anyone holding the permission, not just the original author.
Change statuscreate_incidentsSame gate as commenting — status is treated as part of the ongoing incident-handling permission, not the author-locked original content.
Raise a ticketNone — unrestricted, same as any ticket creationOpens New ticket form with source_incident pre-set to this incident.
Assign the incidentcreate_incidentsNo dedicated assignment permission for incidents (unlike tickets' self/others split) — reuses the same gate as commenting/status changes, since incidents don't have the same self-service queue dynamic tickets do.

States & edge cases

  • Author's shift ends mid-edit — the lock takes effect the moment the shift ends, not when the author next reloads; an in-progress unsaved edit could be lost, so the app should warn before the boundary where practical.
  • Non-author with create_incidents — can comment and change status freely, but never edit the original title/type/details, even immediately after creation.
  • No view_incidents but incident has a linked ticket — per Ticket detail, the ticket shows no indication of the incident at all; there's no partial-visibility state to handle here since this page is simply unreachable.
  • Custom status — a marina-defined addition to the default four; behaves identically to the defaults once configured, no special-casing in the UI.

Mobile vs. desktop

Mobile: content, comment thread, and linked tickets stack vertically. Desktop: comment thread can sit alongside the incident's own content rather than below it, since there's room to keep both visible while replying.

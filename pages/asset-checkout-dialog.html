Assets · Page Specification

Checkout / Return Dialog

Process an asset going out or coming back — one dialog, two modes depending on whether the asset is currently checked out.

Purpose

Reached from Asset detail, gated by manage_assets (checkout records are explicitly within its scope, per Permissions).

Data shown / form fields

FieldTypeNotes
Checking out toContact search/selectDefaults to the current user's own linked Contact (User.contact) but can be set to any Contact, per Data Model — AssetCheckout.
Time out / inDatetime, defaults to nowEditable for backfilling.

Actions

ActionGatingBehavior
Check outmanage_assetsCreates an AssetCheckout: asset, checked_out_by = current user, person = selected contact, time_out = now, time_in = null. Shown only when the asset has no open checkout.
Returnmanage_assetsSets time_in on the open checkout record, then applies Asset.post_return_status — writes a new AssetStatusLog entry (e.g. Jump Pack → "Needs Charging"). Shown only when the asset currently has an open checkout.
CancelNoneCloses without creating or closing a checkout.

States & edge cases

  • Already checked out — the dialog opens directly in Return mode; there's no way to check out an asset that's already out.
  • Returning without a status change — post_return_status always applies on return; there's no option to skip it, since it's the target's configured behavior, not a per-return choice.
  • Person isn't a User — perfectly normal; person is a Contact reference, and most checkouts (e.g. a jump pack lent to a boat owner) go to someone who never signs into the app.

Mobile vs. desktop

Same dialog on both; no meaningful layout divergence beyond the app's general responsive treatment.

Lease Detail

A location's long-term lease record — lessees, dates, variances, documents, and a running comment thread — gated entirely by view_lease.

Purpose

Reached from a Location's detail page (typically a Slip) or a lessee Contact's detail page. This is the long-term counterpart to a short-term Reservation: a marina typically sets a given Slip up as either reservation-enabled (short-term) or leased (long-term), not both — which is exactly what lets a mixed-use Dock keep its short-term Slips and long-term-leased Slips side by side. The data model doesn't hard-enforce that split (reservation_enabled and having a Lease are independent fields), so it's a marina-configuration convention rather than a system constraint.

Data shown

FieldSourceNotes
LocationLease.locationLinks to Location detail; variances are displayed alongside the Location's (and its current Boat's) info there too, per Data Model.
LesseesLease.lesseesRequires view_owner for names, view_contact for phone/email — same gating as everywhere else Contacts appear.
DatesLease.start_date / end_date
Variances & conditionsLease.variances_and_conditionsFreeform text.
DocumentsLease.documentsStored contract files.
CommentsLease.comments → LeaseCommentAuthor, body, timestamp — an ongoing thread, not locked like an Incident's original content.

Actions

ActionGatingBehavior
Create the leasemanage_leaseReached via Location detail's "Add a lease" action when the Location has none yet — this page doubles as the creation form, similar to how Nameless-contact merge handles a not-yet-resolved state. If the Location's reservation_enabled is currently on, shows a warning before creating ("this location currently accepts reservations — continue?") but doesn't block it; proceeding doesn't itself turn reservation_enabled off.
Edit lease detailsmanage_leaseDates, variances, lessees.
Upload a documentmanage_lease
Download a documentview_leaseViewing-tier action, unlike upload.
Add a commentmanage_leaseReuses the same gate as editing, matching the pattern established for Incident comments (create_incidents covers both creating and commenting) — no separate lease-commenting permission.

States & edge cases

  • No view_lease — this page, and any link into it from Location/Contact detail, is unreachable.
  • view_lease without manage_lease — full read access including document download, but no editing or commenting.
  • Expired lease (end_date passed) — shown plainly as historical; the data model doesn't auto-flip the Location's status or reservation_enabled on expiry, so renewing or transitioning a Slip to reservation-enabled after a lease ends is a manual admin action, not an automated workflow.
  • Reservations turned on for this Location while an active lease exists — the symmetric case, triggered from Location detail or Admin → Location Types & Locations setup, not from here; also a warning, not a block, per Data Model — Location.

Mobile vs. desktop

Mobile: fields and documents stack vertically, comment thread below. Desktop: comment thread can sit alongside the lease's own fields, matching the pattern used on Incident detail.

Architecture

API Structure

Two integration surfaces make up the system's API: the frontend talking directly to InstantDB for almost everything, and the frontend talking to the per-marina telephony server for anything that has to touch Twilio.

Two surfaces, one rule of thumb

Frontend ↔ InstantDB

Direct client-side queries, subscriptions, and writes. This covers the entire application except calls and SMS: locations, checklists, incidents, tickets, assets, reservations, contacts, leases, chat, the Activity Log, and Admin configuration.

Frontend ↔ Telephony Server ↔ Twilio

A small REST API, used only for actions that require Twilio: initiating a call/SMS, and receiving Twilio's webhooks. The server writes results into InstantDB; the frontend then reads those results the same way it reads everything else.

Why not route calls through InstantDB writes directly? Because initiating a call or SMS requires a server-side Twilio credential that must never reach the client, and because Twilio's own webhooks need a real HTTP endpoint to call back to. Everything that doesn't have that constraint stays a direct InstantDB interaction — that's the dividing line.

Frontend ↔ InstantDB patterns

PatternUsed for
Subscribed queriesAny list or detail view that should update live — the ticket queue, an active checklist, a chat room, missed-call badge counts.
Local-first writesEvery user action: completing a checklist item, logging an incident, updating an asset's meter. Written locally immediately, synced automatically per Architecture.
Permission-aware rulesInstantDB's permission rules are configured to check the acting user's effective permission (computed per Permissions) before allowing a read or write, so enforcement isn't solely a client-side UI concern.

Because this surface is direct client-to-database, there is no separate application-defined REST or GraphQL schema to document beyond InstantDB's own query/write model and the entity structure already described in Data Model.

telephony server

Telephony Server API

A minimal REST API, scoped entirely to bridging Twilio. Two categories of endpoint: inbound webhooks (called by Twilio) and outbound actions (called by the frontend, authenticated as a logged-in user).

Inbound — Twilio webhooks

EndpointTriggered byBehavior
POST /webhooks/voice/inbound An incoming call to a marina number Looks up which configured line was dialed, applies its routing (e.g. the security line skipping straight to a security menu), performs contact matching, creates a Call record in InstantDB, and returns TwiML directing Twilio how to proceed (ring staff, play a menu, go to voicemail).
POST /webhooks/voice/status Call state changes (ringing, answered, completed, no-answer) Updates the corresponding Call record's status, duration, and missed flag.
POST /webhooks/voice/recording A call recording becomes available Attaches the recording URL to the Call record, if recording is enabled for the marina.
POST /webhooks/voice/transcription A transcription completes Attaches transcript text to the Call record, if transcription is enabled.
POST /webhooks/voice/voicemail A caller leaves a voicemail Stores the voicemail recording URL against the Call record and flags it for the missed-call badge.
POST /webhooks/sms/inbound An incoming text to a marina number Performs contact matching, appends to (or creates) the matching SMSThread, creates the SMSMessage, and marks the thread unread.

Outbound — called by the frontend

EndpointRequiresBehavior
POST /calls/initiate place_calls permission Accepts a target number (or Contact reference) and a line to call from; instructs Twilio to bridge the call to the requesting user's device; creates the outbound Call record.
POST /sms/send place_calls permission Accepts a target number (or Contact reference), a line, and a message body (optionally sourced from an SMSTemplate); sends via Twilio; creates the outbound SMSMessage.
GET /calls/active Authenticated user Returns any call currently ringing/in-progress for the requesting user's line(s) — used to drive the active-call panel/pop-up without waiting on a webhook round-trip to InstantDB in edge cases.
POST /reports/shift/:shiftId/send Authenticated user (the shift's guard, or view_reports) Compiles the shift report from the shift's time window and emails it to the marina's configured recipients (see Architecture — Shift report delivery). Called automatically on shift end; retried on reconnect if the shift ended offline; also usable to re-send.
Auth for the telephony server. Outbound endpoints authenticate the caller using the same session/identity issued by the InstantDB-compatible auth provider (see Architecture), so the telephony server can independently verify the request is from a logged-in user holding the required permission — it does not trust the frontend's own permission check alone.

Contact matching

Both inbound webhook paths run the same contact-matching step before writing a record:

  1. Normalize the incoming phone number and look for an exact match against existing Contact records.
  2. If found, link the Call/SMSThread to that contact.
  3. If not found, create a new Contact with the phone number and no name.

The "nameless contact" prompt and merge-suggestion flow described in Role Workflows happens entirely in the frontend when such a contact is later viewed — the telephony server's job stops at creating the bare record.

scheduling — open question

Scheduled jobs

As noted in Architecture, the mechanism for time- and meter-based checklist/ticket generation is not yet finalized. Whatever mechanism is chosen, it will need its own narrow write path into InstantDB (functionally similar to the telephony server's webhook handlers — evaluate a condition, then create a Checklist or Ticket record) rather than a broad API surface. This section will be filled in once that decision is made.

Architecture

API Structure

Two integration surfaces make up the system's API: the frontend talking directly to InstantDB for almost everything, and the frontend talking to per-marina Twilio Functions for anything that has to touch Twilio.

Two surfaces, one rule of thumb

Frontend ↔ InstantDB

Direct client-side queries, subscriptions, and writes. This covers the entire application except calls and SMS: locations, checklists, incidents, tickets, assets, reservations, contacts, leases, chat, the Activity Log, and Admin configuration.

Frontend ↔ Twilio Functions ↔ Twilio

A small REST API implemented as Twilio Functions, used only for actions that require Twilio: initiating a call/SMS, and receiving Twilio's webhooks. The Functions write results into InstantDB; the frontend then reads those results the same way it reads everything else.

Why not route calls through InstantDB writes directly? Because initiating a call or SMS requires a server-side Twilio credential that must never reach the client, and because Twilio's own webhooks need a real HTTP endpoint to call back to. Everything that doesn't have that constraint stays a direct InstantDB interaction — that's the dividing line.

Frontend ↔ InstantDB patterns

PatternUsed for
Subscribed queriesAny list or detail view that should update live — the ticket queue, an active checklist, a chat room, missed-call badge counts.
Local-first writesEvery user action: completing a checklist item, logging an incident, updating an asset's meter. Written locally immediately, synced automatically per Architecture.
Permission-aware rulesInstantDB's permission rules are configured to check the acting user's effective permission (computed per Permissions) before allowing a read or write, so enforcement isn't solely a client-side UI concern.

Because this surface is direct client-to-database, there is no separate application-defined REST or GraphQL schema to document beyond InstantDB's own query/write model and the entity structure already described in Data Model.

twilio functions

Twilio Functions API

A minimal REST API, scoped entirely to bridging Twilio, implemented as Twilio Functions deployed to the marina's own Twilio account rather than a hosted server (see Architecture — Telephony). Two categories of endpoint: inbound webhooks (called by Twilio) and outbound actions (called by the frontend, authenticated as a logged-in user).

Inbound — Twilio webhooks

EndpointTriggered byBehavior
POST /webhooks/voice/inbound An incoming call to a marina number Looks up which configured line was dialed, applies its routing (e.g. the security line skipping straight to a security menu), performs contact matching, creates a Call record in InstantDB, and returns TwiML directing Twilio how to proceed (ring staff, play a menu, go to voicemail).
POST /webhooks/voice/status Call state changes (ringing, answered, completed, no-answer) Updates the corresponding Call record's status, duration, and missed flag.
POST /webhooks/voice/recording A call recording becomes available Attaches the recording URL to the Call record, if recording is enabled for the marina.
POST /webhooks/voice/transcription A transcription completes Attaches transcript text to the Call record, if transcription is enabled.
POST /webhooks/voice/voicemail A caller leaves a voicemail Stores the voicemail recording URL against the Call record and flags it for the missed-call badge.
POST /webhooks/sms/inbound An incoming text to a marina number Performs contact matching, appends to (or creates) the matching SMSThread, creates the SMSMessage, and marks the thread unread.

Outbound — called by the frontend

EndpointRequiresBehavior
POST /calls/initiate place_calls permission Accepts a target number (or Contact reference) and a line to call from; instructs Twilio to bridge the call to the requesting user's device; creates the outbound Call record.
POST /sms/send place_calls permission Accepts a target number (or Contact reference), a line, and a message body (optionally sourced from an SMSTemplate); sends via Twilio; creates the outbound SMSMessage.
GET /calls/active Authenticated user Returns any call currently ringing/in-progress for the requesting user's line(s) — used to drive the active-call panel/pop-up without waiting on a webhook round-trip to InstantDB in edge cases.
Auth for Twilio Functions. Outbound endpoints authenticate the caller using the same session/identity issued by the InstantDB-compatible auth provider (see Architecture), so the Function can independently verify the request is from a logged-in user holding the required permission — it does not trust the frontend's own permission check alone.

Contact matching

Both inbound webhook paths run the same contact-matching step before writing a record:

  1. Normalize the incoming phone number and look for an exact match against existing Contact records.
  2. If found, link the Call/SMSThread to that contact.
  3. If not found, create a new Contact with the phone number and no name.

The "nameless contact" prompt and merge-suggestion flow described in Role Workflows happens entirely in the frontend when such a contact is later viewed — the Twilio Function's job stops at creating the bare record.

netlify functions

Shift Report & Scheduled Jobs API

Distinct from the Twilio Functions above, these run as Netlify Functions alongside the static frontend (see Architecture — Scheduling and Shift report delivery). Only the shift-report send path touches Twilio, and only as an outbound API call to Twilio SendGrid — it isn't deployed as a Twilio Function itself.

FunctionTriggerBehavior
POST /reports/shift/:shiftId/send Called on-demand by the frontend (authenticated as the shift's guard, or a user with view_reports) Compiles the shift report from the shift's time window and sends it via Twilio SendGrid to the marina's configured recipients. Called immediately on shift end while online; also usable to re-send from the Reports section.
scheduled: shift-report-sweep Netlify Scheduled Function, runs on a short interval (e.g. every few minutes) Finds any Shift marked ended with no report yet sent and dispatches it — the backstop for a shift that ended offline, where the on-demand call never fired.
scheduled: checklist-triggers Netlify Scheduled Function, runs on the marina's finest-configured checklist schedule granularity Evaluates ChecklistTemplates with a schedule-based trigger against the current time and generates a new Checklist instance when due.
scheduled: maintenance-ticket-generation Netlify Scheduled Function, runs daily (time-based rules) and on Meter Reading writes are evaluated inline by the frontend write itself (meter-based rules) Evaluates each Asset's time-based maintenance rules (e.g. "every 90 days since last completed") and creates a Ticket when due. Meter-based rules (e.g. "every 3,000 miles") are checked as part of the InstantDB write when a Meter Reading is recorded, since that's the only moment the value changes.
scheduled: activity-log-retention Netlify Scheduled Function, runs daily Purges Activity Log entries older than the marina's configured retention window, skipping any entry flagged Protected.

Each of these has a narrow write path into InstantDB — evaluate a condition, then create or purge a record — using the same server-side/admin API and per-marina admin token pattern as the Twilio Functions. The shift-report send path additionally holds the marina's Twilio SendGrid API key as a Netlify environment variable, scoped to that one function.

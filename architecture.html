Architecture

Stack & Architecture

One marina, one deployment. No shared infrastructure, no multi-tenant data model to defend — every property gets its own isolated copy of the whole stack.

System overview

Three pieces make up a single marina's deployment:

Static Frontend

The application itself — hosted as a static site on Netlify, on the marina's own subdomain.

InstantDB

The marina's database — local-first, syncing automatically, giving the frontend offline read/write with no custom sync logic.

Telephony Server

A lightweight server bridging Twilio and the marina's InstantDB instance for call and SMS logging.

Per-marina isolation. Each marina receives its own subdomain, its own Netlify deployment, its own InstantDB instance, and its own telephony server (with its own Twilio numbers). There is no cross-marina database, no shared auth realm, and no tenant-scoping logic anywhere in the schema — isolation is structural, not enforced by application code.

Database — InstantDB

InstantDB serves as the system of record and as the offline/sync layer simultaneously. This is the architectural decision that makes "a guard with no signal on a dark dock" a non-event rather than a special case:

  • Local-first reads and writes. The app reads from and writes to a local copy of the data at all times. There is no separate "offline mode" — the app behaves identically whether connected or not.
  • Automatic sync on reconnect. Once connectivity returns, InstantDB reconciles local changes with the server automatically. No custom conflict-resolution, queueing, or retry logic is needed in the application.
  • Realtime updates when online. Given the low concurrent usage this application expects (typically one to two guards active at a time, sporadic maintenance/office usage), InstantDB's built-in realtime and conflict handling is sufficient without additional design.
Design implication. Because sync is automatic, nothing in the data model should assume a strict global ordering of events at write time. Where sequence matters (e.g. an Activity Log), it's derived from timestamps rather than from write order.

Authentication

Auth is handled through an InstantDB-compatible auth provider rather than a bespoke system, so session and identity management stays consistent with the rest of the local-first data layer.

  • Personal devices — a user's phone or tablet stays signed in as that user.
  • Shared devices — some hardware (e.g. a tablet mounted at the security desk) is used by whoever is on shift. The auth flow supports switching the active user on a shared device without requiring a full sign-out/sign-in cycle for routine handoffs, since shift changes happen daily and need to be fast.

The specific provider and the shared-device switching mechanism are an open implementation decision — noted here as a requirement the chosen provider must satisfy, not yet finalized.

Telephony server

A single small server per marina sits between Twilio and that marina's InstantDB instance. Its job is narrow and specific:

  • Receive inbound call and SMS webhooks from Twilio.
  • Route calls per the marina's configured menu (e.g. a secondary line that skips directly to a security queue).
  • Write call/SMS metadata, recordings, and transcriptions (where enabled) into InstantDB as they occur, so the frontend simply reads live data rather than polling Twilio directly.
  • Accept outbound call/SMS initiation requests from the frontend and relay them to Twilio.
  • Perform contact matching (incoming number → existing Contact, or create a new one) as described in Terminology and detailed further in Data Model.
  • Compile and email the shift-end report (see "Shift report delivery" below).

Full request/response shape is covered in API Structure. Architecturally, this server is intentionally thin — it holds no business logic beyond what's required to bridge Twilio events into InstantDB records; everything else (call UI, notes, missed-call badges) is frontend behavior reacting to that data.

Offline & sync behavior

Because InstantDB handles local-first storage and sync natively, no additional offline architecture is required in the application layer. Specifically out of scope for custom engineering:

  • Manual local caching or queueing of writes made while offline.
  • Custom conflict resolution for concurrent edits.
  • Manual "sync now" triggers or sync-status polling — InstantDB's connection state is sufficient for any UI indicator needed (e.g. a small "offline / syncing" badge).

The one area that is not covered by InstantDB's offline behavior is the telephony server, which inherently requires connectivity — call/SMS features are only available while online, and this should be reflected honestly in the UI (see relevant page specs) rather than presented as available offline.

Shift report delivery

The shift-end report is an email, which means it can't be produced by the static frontend alone — something server-side must compile and send it. The natural home is the per-marina telephony server, since it already exists as the one always-on process with a write path into InstantDB; adding a mail-dispatch responsibility keeps the deployment at three pieces rather than four. (The email provider itself is an open implementation choice.)

The flow: ending a shift — by submitting the end-of-shift checklist if one is defined, or by tapping End Shift otherwise — marks the Shift record ended. The server then compiles the report from that shift's window (check-ins, checklists, incidents, tickets, associated by timestamp per the Data Model) and emails it to the marina's configured distribution list.

Offline shift end. A guard may end their shift with no connectivity. The shift-end write succeeds locally like any other InstantDB write; the report is compiled and sent once that record syncs to the server. The report is therefore defined as "sent on shift end, or as soon after as connectivity allows" — never silently dropped.

Scheduling Open question

Several features require something to run on a schedule independent of any user having the app open:

  • Time/schedule-based checklist triggers (e.g. a checklist generated automatically every morning at 6am).
  • Meter- and time-based maintenance ticket generation (e.g. "create a ticket every 3,000 miles" or "every 90 days since last completed").
  • Activity Log retention enforcement — purging entries older than the configured retention window (excluding Protected entries).

Since the frontend is a static site with no persistent server process of its own, this scheduling responsibility needs to live somewhere with an always-on execution context — candidates include a scheduled function alongside the Netlify deployment, a cron responsibility added to the per-marina telephony server, or a dedicated small scheduler service. This is intentionally left open and should be resolved as its own design pass before implementation; noting it here so it isn't lost.

Activity Log generation

The Activity Log (see Terminology) is generated at write time, immutably, by the same code paths that make the underlying change — not reconstructed later from other tables. As a rule of thumb: any user-initiated change to persisted data should produce a corresponding entry, with two categories of exception:

  • Changes to the Activity Log itself (it doesn't log its own writes).
  • High-frequency, low-value changes where logging would create noise without benefit (exact boundaries to be defined per entity in Data Model).

Each marina configures a retention period for Activity Log entries; individual entries can be flagged Protected to survive any retention purge, for legal or evidentiary reasons.
